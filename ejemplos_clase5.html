Arquitecturas Reales a Gran Escala

Analiza a profundidad el diseño de los sistemas de software de líderes tecnológicos globales: **Netflix, Amazon, WhatsApp y Moodle**. Compara cómo aplican y adaptan los estilos arquitectónicos clásicos y modernos para resolver sus retos de negocio únicos.

Análisis Técnico y Propuestas Arquitectónicas

Selecciona una de las siguientes pestañas para explorar el estilo arquitectónico, ventajas, desventajas, diagramación interactiva y propuesta técnica para cada empresa:

Caso 1: Netflix (Distribución masiva de video)

¿Qué estilo utiliza? Microservicios Descentralizados + Orientado a Eventos + Red CDN Propietaria.
Netflix descompone sus capacidades en más de 1000 microservicios independientes, orquestados asíncronamente con Apache Kafka y distribuyendo datos pesados localmente en los ISPs mediante su infraestructura CDN global **Open Connect**.
Principales Ventajas ✔ **Alta Resiliencia**: Los fallos en servicios menores (ej. recomendaciones) no bloquean el streaming básico.
✔ **Escalabilidad Elástica**: Se duplican solo los servidores de streaming y base de datos de reproducción durante picos de tráfico.
Desventajas Identificadas ✘ **Consistencia Eventual Compleja**: La descentralización de datos impide transacciones inmediatas ACID.
✘ **Complejidad en Monitoreo**: Monitorear millones de flujos distribuidos requiere herramientas altamente complejas.
Propuesta Alternativa Para un sistema similar, propondría **Microservicios con Coreografía asíncrona** usando bases NoSQL rápidas (MongoDB) para metadatos, bases relacionales ACID (PostgreSQL) para pasarela de cobros, y enrutamiento perimetral robusto (API Gateway).

Caso 2: Amazon (Comercio Electrónico Descentralizado)

¿Qué estilo utiliza? Microservicios Coreografiados + Descentralización Organizacional de Datos (Database per Service).
Amazon fue pionero en estructurar sus servicios bajo la premisa de "equipos de dos pizzas" (Two-Pizza Teams). Cada módulo funcional (el botón comprar, el carrito, las recomendaciones, el inventario) corre de forma aislada, es dueño absoluto de su base de datos y se expone al resto mediante APIs estrictas.
Principales Ventajas ✔ **Desacoplamiento Organizacional**: Los equipos de desarrollo trabajan de forma independiente sin interferencias cruzadas de bases de datos compartidas.
✔ **Escalabilidad Selectiva**: Se puede escalar masivamente el motor de cobros durante el Black Friday sin tocar el servicio de valoraciones de productos.
Desventajas Identificadas ✘ **Problemas de Consistencia Física**: Al no existir una base de datos centralizada, la sincronización en tiempo real del inventario físico puede sufrir desfases temporales (*consistencia eventual*).
✘ **Dificultad de Pruebas**: Probar el comportamiento integrado localmente de miles de APIs distribuidas es casi imposible.
Propuesta Alternativa Para un e-commerce masivo, propondría **Microservicios con Base de Datos Descentralizada e Integración de Bus de Mensajería Asíncrona (ej. RabbitMQ/SQS)** para sincronizar de manera desacoplada los cambios de inventario y despacho, utilizando bases de datos clave-valor de alta velocidad en memoria (Redis) para persistir carritos temporales.

Caso 3: WhatsApp (Mensajería en tiempo real de alta densidad)

¿Qué estilo utiliza? Cliente-Servidor con Conexiones Sockets TCP Persistentes + Modelo de Actores (Erlang/OTP).
WhatsApp no utiliza HTTP REST tradicional para los mensajes en el chat. Mantiene millones de conexiones sockets TCP abiertas y activas de forma permanente en cada servidor físico. Utiliza el lenguaje **Erlang** para spawnear millones de pequeños hilos (actores) asilados de memoria extremadamente eficientes.
Principales Ventajas ✔ **Concurrencia Insuperable**: Un solo servidor físico puede retener más de 2 millones de conexiones sockets concurrentes activas.
✔ **Bajo Consumo de Hardware**: El procesamiento de Erlang tiene una sobrecarga de recursos de RAM/CPU sumamente reducida.
✔ **Tolerancia a fallos de conexión**: Si un actor de conexión de un teléfono falla, se muere de forma aislada sin afectar a ningún otro usuario.
Desventajas Identificadas ✘ **Curva de Aprendizaje de Erlang**: Es un lenguaje funcional poco común, lo que hace complejo y costoso contratar personal de desarrollo especializado.
✘ **Sincronización Multidispositivo Compleja**: Al no guardar el historial en el servidor central (encriptación end-to-end), sincronizar mensajes entre la PC y el móvil requiere protocolos de red muy complejos.
Propuesta Alternativa Para mensajería instantánea, propondría **Modelo de Actores distribuido utilizando Elixir** (que corre sobre la robusta máquina virtual BEAM de Erlang pero ofrece sintaxis moderna) e implementar **WebSockets persistentes** para la comunicación en tiempo real en web y móviles.

Caso 4: Moodle (Gestión de Aprendizaje Académico Modular)

¿Qué estilo utiliza? Arquitectura Monolítica Modular en Capas (LAMP Stack).
Moodle está escrito principalmente en PHP y se organiza en capas lógicas básicas. Su diseño es modular a nivel de código fuente: posee un Core central que expone APIs y es extendido mediante miles de carpetas de plugins (actividades, formatos de curso, bloques) que se cargan de forma integrada al interpretar la petición HTTP.
Principales Ventajas ✔ **Simplicidad de Despliegue**: Extremadamente fácil de hospedar en servidores de bajo costo y configurar bases de datos únicas.
✔ **Extensibilidad Comunidad**: Miles de plugins listos creados a nivel mundial que se acoplan de forma directa al Core.
Desventajas Identificadas ✘ **Cuello de Botella Transaccional**: Durante exámenes masivos concurrentes, miles de estudiantes escriben en la tabla única SQL al mismo tiempo, colapsando el servidor de BD.
✘ **Falta de Escalabilidad Elástica**: No se puede escalar solo el módulo de "cuestionarios"; se debe replicar todo el código de Moodle de forma monolítica.
Propuesta Alternativa Para un sistema nacional académico, propondría **Arquitectura en Capas Compartidas con Desacoplamiento de BD de Lectura** (Master-Slave Replication), usando bases NoSQL para logs de auditoría masivos y almacenamiento en la nube (ej. AWS S3) para archivos PDF y videos de clases.
